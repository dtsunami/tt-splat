⚡ The short version — what actually changes training today

  • Live & working in BOTH modes (host + device-resident): iterations, save_every (periodic checkpoint), snapshot_every, dashboard_every, Pause/Resume, Save Checkpoint, Focus camera, Prune / Clamp Scale / Reset Opacities, and live per-image mask edits.
  • Startup only: sh_degree, TT_SIZE, TT_MAX_POINTS, TT_TARGET_POINTS — set these before you launch.
  • Densification (NEW — the fix for "stuck at N"): adaptive clone/split/prune is wired in and ON by default (TT_DENSIFY=1). It auto-runs on the densify_from/until/every schedule (live-editable) and there's a manual Densify button. Hard cap = TT_DENSIFY_MAX (default 100k). Works in host and device-resident mode.
  • Profiling: launch with --profile to stream live per-step compute utilization (device-busy % per stage) into the Pipeline-stages telemetry.
  • Still not implemented (tracked in the roadmap): lambda_dssim (the TT loss is pure MSE — a real L1+D-SSIM term is next) and opacity_reset_every (no automatic reset; use the manual Reset Opacities button).

🔧 Operations — the sidebar buttons you click during a run

Prune both modes

Delete every Gaussian whose opacity is below the threshold slider — clears floaters and haze and shrinks the model.
Like erasing the faint smudges a sketch picks up. The slider is the opacity cutoff: 0.005 is gentle, 0.02+ is aggressive. Pairs naturally with Reset Opacities. In device-resident mode the survivors keep their Adam momentum/variance (sliced, not reset).
cmd prunehost ✅ · resident ✅

Clamp Scale both modes

Cap each Gaussian's log-scale (default 2.5) so none can grow into a long needle that streaks across the image.
A haircut for spikes. Reach for it when you see thin bright slivers smearing the render — usually a few overgrown Gaussians.
cmd clamp_scale · max_log_scale=2.5host ✅ · resident ✅

Reset Opacities both modes

Push every Gaussian's opacity back near zero so each one has to re-earn its visibility over the next steps.
A floater amnesty: faint junk that isn't pulling its weight fades out and the optimizer re-grows only what the photos actually need. In CUDA 3DGS this fires automatically every opacity_reset_every — here it's a manual button (automatic reset is a roadmap item).
cmd reset_opacitieshost ✅ · resident ✅

Save Checkpoint both modes

Write the current model to splat.ply immediately.
Works in every mode. Since there's no periodic auto-save yet, this is how you snapshot a good-looking moment before you keep training (or before you tinker with something risky).
cmd savehost ✅ · resident ✅

Pause / Resume both modes

Freeze the training loop between steps without tearing anything down; resume right where it stopped.
Safe to use any time — handy for inspecting the Render | GT | Diff panels, painting a mask, or letting the card cool.
/pausehost ✅ · resident ✅

Focus here both modes

Lock training to one camera so every step optimizes that single view (instead of cycling all of them).
Great for debugging one region — drive the loss on the view you care about, watch it sharpen, then Clear focus to resume the round-robin. Per-image page; shortcut F.
cmd focus_camerahost ✅ · resident ✅

🎚️ Meta-parameters — the editable chips in the run bar (and the Edit modal)

Training

Iterations live-editable

Total optimization steps. iterations
More steps refine detail; too many overtrains into floaters and artifacts. If the render looked better a while ago, lower this. The one config knob you can change live and have it stick.

Checkpoint Interval live-editable

Write splat.ply every N steps. save_every
Periodic checkpoint so a crash or a bad edit can't cost you the whole run. 0 = only at the end. You can also force one any time with Save Checkpoint.

Snapshot Interval live-editable

Save a per-camera render PNG every N steps. snapshot_every
Builds a snapshots/{camera}/step_*.png trail you can stitch into a training-progression video with ffmpeg. 0 = off.

Dashboard Refresh live-editable

Push a live preview every N steps. dashboard_every
Lower = smoother monitoring, slightly slower training (each push reads pixels back from the card).

Quality

DSSIM Weight (λ) not wired

Blend L1 pixel loss with SSIM structural loss. lambda_dssim
In the 3DGS paper the loss is (1−λ)·L1 + λ·(1−SSIM): λ→0 is sharp but noisy, λ→1 is smooth but blurry. The TT loop currently trains on pure MSE — there's no SSIM term — so this knob is inert. Adding D-SSIM is one of the highest-value gaps vs. gsplat.

Opacity Reset Interval not wired

Auto reset opacities every N steps. opacity_reset_every
Stock 3DGS periodically fades everything to force a floater cull. No automatic reset is wired into the TT loop yet — use the manual Reset Opacities button (host mode only) instead.

Densification — wired in & ON by default (the fix for a stuck count)

Start / Stop Step live-editable

densify_from · densify_until
The window during which new Gaussians get added — early enough to fill the scene, stopped early enough to leave time for pure refinement. Defaults 500 / 15000.

Interval live-editable

densify_every
How often, inside that window, the densify pass runs. Lower = more frequent = grows faster. Default 100.

Gradient Threshold live-editable

densify_grad_threshold
Absolute positional-grad cutoff to clone/split a Gaussian — i.e. when the optimizer keeps dragging one blob around because it can't cover the detail. 0 = adaptive (mean+std, recommended). Higher = fewer new Gaussians.

How it grows

clone small high-grad Gaussians, split large ones into two children (scale/1.6), prune floaters — the proven M7 operators. ON by default (TT_DENSIFY=1), capped at TT_DENSIFY_MAX (100k). The manual Densify button fires one pass now.

Set at launch (not in the run bar)

SH degree startup only

View-dependent colour bands (0–3). sh_degree
0 = flat colour; 3 = full specular/reflective sheen (standard). Higher = more parameters per Gaussian. Baked in when the model is initialized — set via config before the run.

Resolution & budget startup only

TT_SIZE · TT_MAX_POINTS · TT_TARGET_POINTS
Training image size, the cap on how many sparse points seed the model, and an optional densify-the-seed target for scale tests. Environment variables — see Modes & env vars.

📊 Metrics — the read-outs in the stats bar & loss chart

Loss / MSE

Mean-squared pixel error between render and ground truth.
This is the number the optimizer is actually driving down. Lower is better; watch the blue curve in the chart fall and flatten. Note: the chart's loss and mse series are the same value — mse is fed loss verbatim, so they overlay.

L1

Mean absolute pixel error.
A gentler companion to MSE — less dominated by a few very-wrong pixels. Lower is better.

PSNR

Peak signal-to-noise ratio in dB between render and ground truth.
Goes up as quality improves (≈30 dB good, ≈40 dB great). Correctly labeled now — this field used to be mislabeled "SSIM". A real structural-similarity (D-SSIM) loss term is a roadmap item.

Gaussians

Live count of Gaussians in the model.
Shrinks when you Prune; would grow with densification once that's wired in. A rough proxy for model capacity / file size.

🧭 Modes & environment variables — how much of the loop runs on the Blackhole

Three execution modes (set before launch)

  • Host autograd (default): the verified reference render + torch autograd on the CPU. Correct and flexible — every Operation works — but the slow path. All the structural commands (Prune, Clamp, Reset) are honored here.
  • Device render (TT_DEVICE_RENDER=1): the dashboard preview frame is rasterized on the Blackhole (M14); gradients still come from host autograd.
  • Device-resident (TT_DEVICE_RESIDENT=1): the full inner loop — projection, raster, both backward passes, and Adam — runs on the card with params + optimizer state resident. This is the scale path (1600 px, 50k+ Gaussians) and the one that lights up the Pipeline stages bar. It skips Prune / Clamp / Reset / set-LR (params never come back to host in the inner loop).

🫧 Pipeline stages — the colored bar under the preview (device-resident only)

Each training step is a round trip through these stages; the bar shows how many milliseconds each one took on the Blackhole this step, so the tallest segment is your current bottleneck.
