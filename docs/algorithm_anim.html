tt-splat: the on-device 3DGS training loop 🫧🎢

One training step is a round trip on the Blackhole. Fuzzy 3D Gaussians live in GDDR/L1 and stream through a grid of Tensix cores: project to 2D, bin into screen tiles, alpha-blend to an image, measure the error, then push every blob a little less wrong — and the optimizer never leaves the card. Seven stages, looping C → B on resident params.

The seven stages (the colored bar above maps 1:1 to the training page):
  • B · project fwd device — batched ttnn over N: mean@R+t, conic, SH→color, sigmoid opacity. 3D blobs → 2D ellipses.
  • bin · tile + sort host glue — 3-sigma AABB → per-Tensix-tile gaussian lists, sorted front-to-back by depth.
  • raster · blend device (M14) — per Tensix core, front→back alpha-blend on the SFPU; outputs image + final transmittance T.
  • loss host — MSE + dL/dimage, masked, elementwise.
  • A · raster bwd device — one grid dispatch, all tiles in parallel; per-pixel gradient products. Historically the bottleneck.
  • D · project bwd device — analytic: conic→Σ2D→scale/quat, uv→mean, color→SH, op→logit. Dominant on real data.
  • C · Adam device — resident param update over N; params + momentum stay on the card, then loop back to B.
The grey host glue sliver = bin (the UI also folds in loss, but loss isn't emitted, so it's effectively bin only). Everything colored runs on the Blackhole. Watch the bar: each time we speed up the tallest stage, a different one becomes the bottleneck.
