How tt-splat trains, for humans 🎢

We're teaching a cloud of fuzzy 3D blobs (Gaussians) to look like a real photo. Each training step does a round trip: paint a guess (forward), see how wrong it is, then push every blob a little less wrong (backward). Five stages do this, and all five run on the Blackhole — the blobs and the optimizer never leave the card. The colored bar on the training page is these exact five. Click a stage below to jump to it.
▶ Forward — paint a guess

B · project forward device

Turn each 3D blob into a 2D ellipse on the camera's screen: where it lands, how stretched it is, its color and opacity.
Like a slide projector — a 3D thing on the carousel becomes a flat shape on the wall. We do the camera math (and spherical-harmonic color) for every blob at once.
runs on device · reads the resident params

raster · blend device

Walk the ellipses front-to-back and blend their colors into pixels until the view is opaque. This is the actual rendered image.
Like stacking colored glass: each blob tints what's behind it a little, and once it's solid you stop. The R/G/B channels run side-by-side on three bands of cores.
runs on device (M14) · outputs image + transmittance
◀ Backward — learn from the mistake

A · raster backward device · biggest

Given "the painting is wrong here," figure out how much each blob's 2D ellipse is to blame — the gradient of the blend.
Replay the glass-stacking in reverse, asking each pane "if you'd been a bit different, would the picture be closer?" It's the heaviest stage, so it's where we fight hardest for speed.
runs on device · the current bottleneck

D · project backward device

Carry the 2D blame back through the camera math into the real 3D knobs: position, scale, rotation, color, opacity.
Run the slide projector backward — "to fix the shape on the wall, nudge the 3D thing this way." These 3×3 camera matrices are the next stage we're moving onto the idle matrix engine.
runs on device · reuses B's saved work

C · Adam device · tiny

Actually nudge every blob's knobs using the gradients — with momentum and per-knob caution so it converges fast and smooth.
The smart coach: remembers which way things have been improving and how jittery each knob is, then takes a confident step. → full Adam cartoon
runs on device · params + momentum stay resident
The grey host sliver on the bar is glue that still runs on the CPU — binning/sorting the blobs into screen tiles, and computing the pixel error. Everything colored is on the Blackhole. A (raster backward) is the tallest bar, so it's the bottleneck; we just made raster and A ~1.7–1.9× faster by running R/G/B in parallel on separate core-bands, and D (projection) is next — those 3×3 matrices are literally matrix-multiplies for the idle matrix engine. Watch the bar: every time we speed up the tallest stage, a different one becomes the tallest.
