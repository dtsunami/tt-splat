Adam, for humans 🎢

A ball rolls downhill on a loss landscape (darker = better). Training = find the lowest point. Plain gradient descent (orange) zig-zags across the narrow valley. Adam (cyan) adds two memories — momentum and per-direction caution — and glides straight down. This is the exact algorithm running on the Blackhole as our device-resident optimizer (Stage C).

How to read it. The red arrow is the raw gradient (noisy). The blue arrow is momentum m — the average of recent gradients, so it cancels the side-to-side jitter and points down the valley. v (yellow) quietly measures how steep each axis has been; Adam divides the step by √v, so the steep axis (across the valley) gets tiny steps while the flat axis (along the valley) gets big ones — that’s why cyan goes straight where orange bounces.

Why it oscillates around the minimum. Adam normalizes by √v, so each step is about lr-sized regardless of how small the gradient gets — near the bottom it keeps taking ~lr steps and hovers/orbits instead of stopping (the noise keeps v alive, which feeds the jitter). That's real Adam, not a bug. The fix is an adaptive lr schedule: adaptive lr (anneal) decays lr ∝ 1/√t so the step shrinks over time and the ball settles in — it's on by default; uncheck it to watch the orbit return.

Knobs: ε is Adam's real epsilon — tiny = fully adaptive per-axis; crank it up and the √v normalization washes out (Adam → plain momentum). noise is the gradient jitter (a stand-in for mini-batch noise) — set it to 0 for a clean gradient and watch cyan glide dead-straight. Also try β₁ to 0 (no momentum → jitter returns) or β₂ to 0.9 (short caution memory → less adaptive).
