# tt-splat — Performance & Data-Locality execution plan

> Recovers & supersedes the lost **M2 local-accumulation** intent (`docs/pathclear/m2_local_accum.py`, `docs/pathclear/m2_scatter_gather.py`, `docs/FEASIBILITY.md`/`docs/PROGRESS.md`). Produced by a 38-agent grounded workflow (map → mine → adversarially verify → synthesize); citations are file:line in the live tree. Acceptance bar shifts from bit-exact (host fp64) to the **2e-2 grad gate + multi-step loss-curve parity** once any on-device reduce is introduced.

## Executive summary

Ground truth (measured, N=1024/96px/warm): step=484ms = B13 + raster_fwd35 + A400 + D25 + C3. Stage A is the wall and it is HOST overhead, not device compute: the m17 kernel runs ~10ms; the other ~390ms is readback (~260ms, ~21*nbatch tiny to_torch at fused_backward.py:121), arg-pack (~55ms, host f2u loop at fused_backward.py:104-110), and host scatter-add (~12ms at fused_backward.py:123-131). nbatch=63 in the synthetic worst case (all Gaussians cluster into ~one tile); every host cost scales with nbatch. The deepest root cause: the per-Gaussian grad = sum over the tiles its footprint replicates across, and the destination [N] slot is looked up in the HOST-resident bin/sort index (tile_lists, device_resident.py:106-107), which forces the readback to exist.

The plan attacks this in leverage+dependency order. The single highest-leverage primitive is MISSING and everything depends on it: an on-device per-tile spatial reduce. Today m17 WRITER writes K full 32x32 product tiles per output (m17:128-139) and the spatial sum is done on host (fused_backward.py:121, m17:11 'reduce=host'). Build that reduce, then collapse the per-chunk drain, then move the cross-tile reduce on-device via the PROVEN owner-single-writer pattern (m2_scatter_gather: 8 src x 512 partials, FP32, bit-exact, no atomics, drain at line 53 sums floats directly — so FP32 is correct, int32 is only a perf option). Hash-home (home(g)=g%ncores) replaces the host sort-index lookup with a computed destination. Cheap host-side wins (channel fusion, arg-pack-once, FUSED_K bump) bank 100-150ms while the resident-kernel substrate is built. The scale/host-elimination endgame (GDDR tiering, x280 sort, wave overlap) is correctly deferred: it has near-zero value at the measured N and depends on the prior stages.

Decisive de-prioritizations from the verdicts: drop the partial per-row reduce (precision rationale is inverted), drop standalone FUSED_K=64 (blows L1: 7*64*4KB=1.79MB > 1.5MB), drop standalone barrier-removal/wave-overlap and x280-in-loop (zero standalone value + NoC0 hang hazard on a single-owner training chip), down-rank GDDR tiering (only matters past ~2.4M Gaussians, far above measured 65536).

## Data-locality principles (the design rules this plan enforces)

- Compute reduces BEFORE data crosses a boundary: a per-Gaussian grad is a spatial sum over its 32x32 tile pixels (intra-core, zero cross-core traffic) and a sum over the tiles it replicates across (cross-core). Do BOTH on-die; never ship 1024x the volume to host to do a sum the engine can do where the data lives.
- Owner-single-writer, destination COMPUTED not looked up: replace the host scatter keyed by the bin/sort index with home(g)=g%ncores, slot=g//ncores. Each source NoC-writes into a DISTINCT per-source inbox slot in the owner's L1; the owner reduces as the only writer. Proven contention-free and bit-exact on silicon (m2_scatter_gather), no atomics, no noc_accumulate, no wedge.
- Decouple tile-MEMBERSHIP (raster replication, fundamental — a Gaussian's pixels span tiles) from accumulator-OWNERSHIP (a computed hash). Raster keeps its block-shard-one-tile-per-core layout; only the gradient destination changes.
- Params resident, grads resident: params + Adam m/v never leave the card (already realized, device_adam.py:26-28; 23ms vs 201ms Adam at N=65536). Extend residency to the per-step grad path so A->D->C never round-trips host.
- Allocate accumulators ONCE and reduce in place: mirror PXt/PYt (allocated once, reused across all k,c at fused_backward.py:65-66). Stop the per-chunk realloc of 7*64KB outs (fused_backward.py:91).
- Only the deferred bin/sort crosses host — and ONLY until it can move on-die. Everything else (projection, raster, accumulate, Adam) stays resident; the forward 2D readback exists solely to feed the host lexsort and is the last host hop to remove.
- Encode args once per immutable thing, stream the variable slice: per-Gaussian scalars are step-invariant except col (per-channel); pack once, don't rebuild RuntimeArgs per (chunk,channel,core).

## Exploration track — RUN FIRST (de-risk the XL bets before building)

### E1. Split the 260ms readback into per-call latency vs per-byte transfer/untilize
- **Question:** Is the Stage A readback latency-bound (call count) or bandwidth-bound (volume)? This decides whether the on-device per-tile reduce (cuts bytes) or the per-chunk/channel drain collapse (cuts calls) is the bigger lever, and bounds every readback win.
- **How:** FB_PROF=1 already times the readback bucket (fused_backward.py:82-83). Add a microbench in scratchpad: to_torch a [1536,96] block-sharded L1 tensor (one current output) M times, and a single 32x32 tile N times; compare per-call ms. Effective BW = 780MB/260ms = ~3GB/s is far below PCIe gen4 (~25GB/s), strongly implying latency/untilize-bound, but measure to confirm the ~21*nbatch call-count is the real driver.

### E2. On-device reduce primitive smoke test
- **Question:** Can reduce_tile<SUM,REDUCE_SCALAR> with a scaler CB of 1.0 collapse a 32x32 fp32 tile to a scalar within the 2e-2 grad gate, given m17 CANNOT enable fp32_dest_acc (uses all 16 bf16 dst slots, m17:10)? enforce_fp32_accumulation in reduce requires DST_ACCUM_MODE — check if it is usable in a SEPARATE compute pass that does not carry m17's dst state.
- **How:** Standalone generic_op (a scratchpad smoke_reduce): one COMPUTE kernel reads a known 32x32 tile from a CB, reduce_init + reduce_tile(SUM,REDUCE_SCALAR) + pack scalar, compare to host fp64 .sum(). Sweep magnitudes incl signed products (a/b/c grads have dx*dy sign flips). Decide: fp32-accum reduce in a 2nd pass vs bf16. This de-risks the load-bearing primitive before any kernel surgery.

### E3. Owner-reduce FP32 at realistic fan-out
- **Question:** Does the m2_scatter_gather owner-single-writer pattern stay bit-exact and avoid NoC wedge when scaled from 8 src->1 home to a dense ~all-cores->~all-owners all-to-all with per-(source,gid) distinct slots? And is FP32 drain fast enough that the reduce is not a new bottleneck (it accumulates floats directly at m2_scatter_gather.py:53)?
- **How:** Extend docs/pathclear/m2_scatter_gather.py to 7 grad scalars per gid, N_SRC up to the raster grid count, each source writing to home(g)=g%ncores at a distinct slot. Verify acc==expected (FP32). Time the drain cycles/elem (already reported at m2_scatter_gather.py:110). If FP32 60-cyc/elem is too slow at scale, fall back to int32 x8-unroll (3.09 cyc/elem, proven) with a per-group fixed-point scale — but only if FP32 fails the budget, since FP32 is bit-exact and dodges quantization risk.

### E4. Inbox sizing under worst-case replication
- **Question:** What is the max per-owner inbox depth = (max tiles any owned Gaussian straddles) x (sources), and does it fit the 1.5MiB/core L1 alongside CBs + raster shards at target N? bin_sort benches 10-25x duplication at 1024px (bin_sort.py:100).
- **How:** Instrument bin_and_sort to log per-Gaussian tile-count distribution (counts array, bin_sort.py:33) on the corgi scene and the synthetic worst case. Compute worst-case bytes/owner = max_replication * 7 grads * 4B * gaussians_per_owner. Confirm <512KB/core headroom; if not, this gates the GDDR-tiering stage.

### E5. Persistent-kernel dynamic-arg streaming feasibility
- **Question:** Can a single generic_op COMPUTE kernel loop over ALL chunks of its tile internally, reading each chunk's <=16 Gaussian param words from a per-core L1 scratch buffer (not get_arg_val runtime args, which are bound per-launch and capped at max_runtime_args=341 ~= 42 Gaussians)? This is the substrate for collapsing nbatch*3 dispatches to ~3.
- **How:** Scratchpad prototype: READER DMAs a host-staged [chunks*16*8] f2u param buffer from L1, COMPUTE reads scalars from a CB instead of get_arg_val, S/T CB recurrence extended past chunk boundaries (the recurrence already works within the K-loop, m17:40-52). Verify bit-exact vs the per-chunk path on a single tile. This de-risks whether 'resident chunk loop' is buildable on the ttnn generic_op stack at all before committing to the XL integration.

## Staged plan

### Stage 1 — Bank the cheap host-side Stage A wins (arg-pack-once + channel-fuse arg path + alloc-once)  ·  *effort S (hours)*
- **Goal:** Cut 80-130ms off the 484ms step with pure host-side refactors, no kernel changes, no precision risk — banked immediately and independently.
- **Mechanism:** (a) Pack per-Gaussian COMPUTE args ONCE per step: precompute a per-gid f2u tuple for the 7 channel-invariant geometry words (cx,cy,a,2b,c,op,b) and substitute only the col word per channel, replacing the per-(chunk,channel,core) f2u loop (fused_backward.py:104-110). MUST use a dict/list-extend path, not numpy gather+.tolist() (measured slower). (b) Hoist the 7 outs + Sout/Tout allocation OUT of the chunk loop where possible (fused_backward.py:91-93) — note buffer addresses change per chunk so reader args can't be fully hoisted, but the alloc churn (7*64KB/core/chunk) can be reduced by reusing buffers. (c) Short-circuit empty tiles (reuse a cached dummy cargs block) so the worst-case nbatch=63 stops paying padded arg-pack for 8/9 idle cores.
- **Locality exploited:** Temporal locality of the arg payload: per-Gaussian scalars are step-invariant (only col is per-channel); the host redundantly re-serializes immutable data nbatch*3*ncores times.
- **Expected win:** arg-pack 55ms -> ~16-20ms (measured ~3x in Python microbench at nbatch=63); plus alloc-churn and empty-tile savings. Net ~35-50ms/step (~7-10%). Largest in the clustered worst case.
- **Verification gate:** FB_PROF=1 args/alloc timing before/after; geomg/colg bit-exact vs current path (identical f2u bytes, just built once) on the corgi + synthetic scenes.
- **Dependencies:** none
- **Risks:**
  - col is per-channel (fused_backward.py:106 colv[k][i]) — treating it as step-invariant silently corrupts color grads; pack geometry once, col per channel
  - reader buffer addresses change per chunk (outs realloc) so reader args can't be hoisted to per-channel unless alloc is also hoisted

### Stage 2 — On-device per-tile spatial reduce (the missing load-bearing primitive)  ·  *effort L (days)*
- **Goal:** Collapse each 32x32 grad product tile to a scalar IN the kernel so the WRITER emits 7*FUSED_K scalars/core instead of 7*FUSED_K*1024 floats — ~1024x readback volume cut and deletes the host .sum(dim=(2,4)) (fused_backward.py:121). This unblocks Stages 3, 5, 6.
- **Mechanism:** Add a SECOND compute pass (separate or appended): pack each grad product (d3..d9) to a temp CB, then reduce_init + reduce_tile<SUM,REDUCE_SCALAR>(CB->dst) + reduce_uninit per output, packing one scalar into a contiguous K-scalar layout. Restructure WRITER (m17:132-138) to write K scalars/output into one tile, and shrink the outs _block allocation (fused_backward.py:91) + host reshape to [GY*FK, GX]. reduce reads from a CB, NOT from dst (the grads live in dst and are never spilled) — so this is a pack->CB->reduce->pack restructure, not an in-place dst reduce. Use enforce_fp32_accumulation if the smoke test (exploration probe 2) shows bf16 reduce fails the gate.
- **Locality exploited:** The per-Gaussian grad reduction is entirely intra-tile: all 1024 pixels live in the producing core's own L1/dst. Reduce where the data lives; never spill the full 32x32 to PCIe.
- **Expected win:** Readback BYTES collapse ~1024x (780MB/step -> ~760KB) and the host .sum() (~78-156ms of CPU touching 780MB) disappears. Realistic standalone ~100-180ms saved IF the readback has a meaningful per-byte/host-sum component (confirm via exploration probe 1); the per-call latency floor (~21*nbatch calls) remains until Stage 3.
- **Verification gate:** m17 grad-check harness (run() K=2/K=4, worst_err/scale < 2e-2, m17:207) AND bit-compare fused_backward_grid geomg/colg vs the current host sum(dim=(2,4)) on the N=1024 synthetic scene. Re-verify precision: bf16 1024-term reduce vs host fp64 — must stay under 2e-2 (current headroom is 8e-3->2e-2, ~2.5x).
- **Dependencies:** exploration: on-device reduce primitive smoke test, exploration: readback latency-vs-bandwidth split
- **Risks:**
  - reduce_tile reads a CB not dst; m17 grads are in dst regs — requires pack->CB->reduce->pack restructure per output (~4x more ops/output/Gaussian), effort closer to L
  - fp32_dest_acc cannot be enabled in m17 (halves dst 16->8, all 16 used); reduce may run bf16 -> 1024-term signed sum (a/b/c grads flip sign) could exceed 2e-2 gate
  - reduce_init reprograms packer edge masks; must bracket with reduce_uninit and not corrupt the existing pack sequence; the API is flagged for churn

### Stage 3 — Collapse the per-chunk drain: accumulate on-device, drain ONCE per channel  ·  *effort L (days)*
- **Goal:** Cut the ~21*nbatch to_torch CALL COUNT to ~7 (or fewer) per channel — attacks the latency-bound residual of the 260ms that Stage 2 leaves behind.
- **Mechanism:** Build the resident chunk-loop substrate (exploration probe 5): the m17 COMPUTE loops over ALL chunks of its tile internally; S/T recurrence already lives in L1 across the K-loop via CB26/27->CB24/25 (m17:40-52) and across chunks via the host rebind St,Tt=Sout,Tout (fused_backward.py:133) — extend it past chunk boundaries with a per-core L1 count word so each tile reads its own chunk count. Stream each chunk's <=16 Gaussian param-bytes from a per-core L1 scratch region (NOT runtime args — capped at max_runtime_args=341 ~42 Gaussians). Accumulate the reduced scalars (from Stage 2) into a per-core L1 accumulator, drain once per channel. Allocate accumulators once (mirror PXt/PYt, fused_backward.py:65-66).
- **Locality exploited:** The chunk loop is ALREADY device-resident dataflow (S/T threaded in L1, zero host between K-iterations); the ONLY host re-entry is the per-chunk grad harvest. Keeping the whole tile's Gaussian stream on-core eliminates (nbatch-1) round-trips/core.
- **Expected win:** After Stage 2 made transfers tiny, collapsing nbatch*3 drains to ~3 reclaims the per-call latency floor. Combined with Stage 2, readback ~260ms -> low tens of ms; also removes per-chunk alloc churn. Standalone slot-keyed accumulation is INFEASIBLE (L1 blowup: nbatch=63 of un-reduced outs = 27MB/core; and slot reuse across disjoint chunks double-counts) — this stage REQUIRES Stage 2's scalars and a Gaussian-keyed (or per-tile-then-summed) accumulator.
- **Verification gate:** Bit-compare per-Gaussian grads (drain-once path) vs current per-chunk path on corgi + synthetic; m17 2e-2 gate; end-to-end grad-equivalence vs last_g3 (device_resident.py:152). Single-tile prototype bit-exact before grid integration.
- **Dependencies:** Stage 2 (on-device reduce — scalars must exist to accumulate), exploration: persistent-kernel dynamic-arg streaming feasibility
- **Risks:**
  - resident-kernel + L1-param-streaming is a new dataflow on the ttnn generic_op stack — must be prototyped (probe 5) before committing
  - load imbalance: collapsing to one dispatch serializes the hot tile's full Gaussian stream on one core while others idle; wall-clock bounded by busiest core (worst at nbatch=63)
  - cross-chunk slot reuse double-counts unless keyed by Gaussian/owner — couples this to Stage 4 for correctness at scale

### Stage 4 — Channel fusion: one dispatch per chunk instead of three  ·  *effort L (days)*
- **Goal:** Amortize the channel-independent geometry work and cut dispatch/readback/arg-pack count ~2.3-3x for Stage A.
- **Mechanism:** Restructure m17 COMPUTE to compute the channel-INVARIANT geometry factors (dx, dy, dx*dx, gexp, alpha, rec, T_run) ONCE, then run 3 sequential color sub-passes reusing dst, each consuming its own dLdC tile + col scalar and emitting col grad x3 plus the geometry grads summed over channels. The geometry GRAD VALUES are channel-dependent (base=dLda*alpha, S is color-weighted, m17:100-105) so they must be summed across channels, but the geometric FACTORS are shared. Triple the per-channel CBs (dLdC, S-recurrence) keeping T_run single; outputs become 6 geom + 3 col = 9.
- **Locality exploited:** Per-Gaussian geometry (dx,dy,gexp,alpha) is recomputed in registers 3x today; computing once and reusing across channels keeps the working set in dst/CB and removes 2/3 of the redundant compute + the per-channel arg-pack walk (fused_backward.py:85 k-loop re-runs the gx/gy pack 3x).
- **Expected win:** arg-pack ~3x (the gx/gy walk re-runs per channel today); readback CALL count 21*nbatch -> 9*nbatch (~2.3x, not 3x, since col-grad triples outputs); dispatch nbatch*3 -> nbatch. After Stages 2+3 this is a clean ~2.3x multiplier on an already-small readback. Best framed as a follow-on amortizer, not standalone.
- **Verification gate:** Per-channel col grads bit-compared (err/scale<2e-2) against the current 3-pass output before deleting the 3-pass path; full grad-equivalence vs last_g3.
- **Dependencies:** Stage 2 (reduce) and Stage 3 (drain-once) — the multiplier only pays on the collapsed readback
- **Risks:**
  - dst exhaustion: m17 uses all 16 bf16 slots; cannot hold 3 channels concurrently -> must be 3 sequential sub-passes sharing CB-staged geometry, adding copy_dest/CB round-trips and bf16 rounding
  - S-recurrence is color-weighted (per-channel) — easy to conflate with the channel-independent T_run and silently corrupt grads

### Stage 5 — Hash-home owner-local on-device gradient reduce (remove the host scatter + the A->host->D re-upload)  ·  *effort XL (week+)*
- **Goal:** Eliminate the host scatter-add (~12ms) and the redundant device->host->device hop (Stage A scatter -> Stage D _t2t re-upload, device_project_backward.py:93), and remove the host-resident sort-index from the BACKWARD accumulation path. This is the architectural core.
- **Mechanism:** After Stage 2's per-tile reduce, each raster-backward core holds <=FUSED_K per-(tile,slot) grad scalars. The WRITER NoC-writes each Gaussian's 7 scalars into its OWNER core's inbox at COMPUTED home(g)=g%ncores, slot=g//ncores, into a DISTINCT per-source slot (m2_scatter_gather.py:34 base=inbox+src_idx*k*4 — no atomics). Each owner reduces its inbox as single writer (m2_scatter_gather.py:53, FP32 float accumulate, proven bit-exact). The gid is known per slot (it is in tile_lists already used for arg-pack). The reduced [N] grad flows straight into Stage D as ttnn (project_backward already accepts ttnn upstreams, device_project_backward.py:93 isinstance check) — the from_torch re-upload vanishes; grads stay resident A->D->C (mirrors the proven 23ms-vs-201ms Adam win).
- **Locality exploited:** Decouples tile-membership (raster replication, fundamental) from accumulator-ownership (computed hash). Converts a host global scatter keyed by the bin/sort index into a hash-partitioned, contention-free, owner-local L1 reduce (single writer). 28B/Gaussian fits L1 (~18.7k Gaussians/512KB at 1.5MiB/core).
- **Expected win:** Removes ~12ms scatter + the D re-upload (slice of D=25ms) + the structural reason for the host harvest. Combined with Stages 2-4, Stage A 400ms -> roughly 60-90ms (kernel ~10ms + reduce + NoC scatter + still-host arg-pack of gids). Path to <30ms only after bin/sort moves off host (Stage 7).
- **Verification gate:** Extend m2_scatter_gather.py to the real 7-grad, gid-keyed, multi-owner case and verify acc==expected in FP32 (bit-exact) BEFORE integration (exploration probe 3). Then end-to-end grad-equivalence (last_g3 vs host autograd) + multi-step loss-curve parity on corgi.
- **Dependencies:** Stage 2 (scalars must exist — shipping 32x32 tiles over NoC to owners would be WORSE than today), Stage 3 (resident drain-once substrate), exploration: owner-reduce FP32 at realistic fan-out, exploration: inbox sizing under worst-case replication
- **Risks:**
  - gid still host-packed per dispatch from tile_lists -> the ~55ms arg-pack is MOVED not removed until bin/sort moves on-die (Stage 7)
  - all-to-all noc_inline_dw_write storm at ~130x130 is untested vs m2's 8->1; BH all-4-mem-ports inline-write constraint + NoC hang hazard -> validate fan-out before integration
  - replicated gids: multiple sources write the same gid -> owner must SUM (segmented/indexed reduce), needs per-(source,gid) distinct slots; inbox sizing for 10-25x dup must fit L1
  - FP32 reduce is bit-exact (proven) so PREFER it; int32 fixed-point is a perf fallback only, and quantization would break the bit-exact gate -> use FP32 unless the fan-out benchmark forces int32

### Stage 6 — Persistent L1-resident owner accumulators + fuse B/D projection dispatch  ·  *effort XL (week+)*
- **Goal:** Make the owner accumulators survive the inner loop (no per-step alloc/free) and collapse the ~440 projection elementwise dispatches (B=13ms+D=25ms, dispatch-bound) toward O(1), as the on-die residency layer for scale.
- **Mechanism:** (a) Allocate per-owner L1 accumulators as RAW byte buffers (NOT ttnn TILE_LAYOUT — 4KB tile granule defeats 28B/Gaussian packing; use the m2_local_accum raw volatile-pointer pattern) on a dedicated owner band, addressed by home(g)/slot via an explicit physical-core lookup table (the worker grid has router-only column gaps, so g%ncores must map through a table, not a rectangular CoreRange). (b) Fuse the B+D projection op-graphs into one (or a few) resident generic_op compute kernels keeping the ~33 geom + ~K+8 color intermediates in L1/registers instead of ~440 GDDR write+read pairs. Camera scalars (device_project.py:48-51) are the only per-step variable input.
- **Locality exploited:** Persistent L1 residency is the thing currently missing everywhere (raster accC/accT freed at render_device.py:129; Stage A outs realloc'd per chunk). Projection intermediates round-trip GDDR per op today; fusing keeps the working set on-core per tile-shard.
- **Expected win:** Removes per-step accumulator alloc/free; projection 38ms -> ~25-32ms (dispatch overhead removed; win grows with N as GDDR round-trips dominate). Enables Stage 5 to stay 100% on-die for the measured regime with headroom (~2.4M Gaussians at 512KB/core).
- **Verification gate:** Allocate-and-readback test of the raw L1 acc across owner cores (extend m2_local_accum to ncores), confirm no L1 OOM with raster+CBs co-resident; fused B/D outputs bit-compared vs current project_geom/color/op and project_backward g3 at err/scale gate; full grad-equivalence.
- **Dependencies:** Stage 5 (hash-home — the accumulators need owners and a delivery path)
- **Risks:**
  - raw-L1 addressing with computed home/slot is silent-corruption-prone if a slot overruns a core's shard (no bounds check in a bare kernel); an out-of-shard NoC write near router/ARC tiles risks the NoC0 hang
  - fp32 projection needs precision (conic inversion det=ac-b^2, z-clamp); 8 fp32 dst slots force hand-scheduled L1 spilling of dozens of live tensors — large correctness surface, re-derive the analytic backward against the fused substrate
  - true usable core count must come from device.compute_with_storage_grid_size at runtime, not the 140 descriptor template

### Stage 7 — ENDGAME (deferred): on-die bin/sort + GDDR owner-partitioned tiering + wave overlap  ·  *effort XL (week+)*
- **Goal:** Remove host from the inner loop entirely and scale past the L1 budget — but ONLY after Stages 1-6 land and ONLY when N or scene dynamics justify it. Near-zero value at the measured operating point.
- **Mechanism:** (a) Move the embarrassingly-parallel bin/sort parts on-device (cull mask zc>1e-4, var_x/var_y from conic, AABB tile-range = ttnn elementwise) keeping results resident; the genuinely sort-shaped lexsort is Tensix-hard (no radix sort) — defer it; consider a counting-bucket on-device sort BEFORE any x280 path. (b) GDDR owner-partitioned [N] grad stripes (single writer per region, mirrors m2 inbox exclusivity at off-chip scale) for N beyond the ~2.4M L1 ceiling; L1 holds the hot window, owner RMW-adds then flushes its stripe. (c) Wave overlap: once stages are resident + host-free, replace the inter-stage synchronize_device barriers (device_resident.py:90,121,141,153,158) with on-NoC semaphore handoff between functional core bands, double-buffering raster k / owner-reduce k-1 / projection k+1.
- **Locality exploited:** Closes the last cross-stage host hop (the forward 2D readback exists solely to feed host bin/sort); owner-partitioned GDDR keeps every off-chip grad write conflict-free; band overlap reclaims device idle once host is out of the loop.
- **Expected win:** At measured N: ~0 (does not touch the wall; host bin lexsort is sub-ms at N=1024). At millions of Gaussians: capacity + host-elimination enabler. Wave overlap trends per-step toward max(band) not sum(bands) but is the LAST increment, gated on everything prior being host-free.
- **Verification gate:** On-device cull/var/AABB bit-exact vs host bin_and_sort outputs; counting-bucket/x280 sort permutation bit-identical to np.lexsort (incl tie-breaks) before wiring; GDDR-stripe path grad-equivalence at a synthetic N that forces L1 spill; wave-overlap gated on loss-curve parity (bit-exactness not required across reordering, parity is).
- **Dependencies:** Stage 5, Stage 6
- **Risks:**
  - x280 harts are still in reset (bringup never completed; wedged NoC0 once); register access to L2CPU hangs NoC0 recoverable only via tt-smi -r 0 — do NOT put x280 in the per-step loop on a single-owner training chip until bringup is solid AND a non-x280 counting-bucket sort is ruled out
  - host bin/sort becomes the real scale-out wall first (~6s/step at 2.4M Gaussians * 15 dup at 6 M-inst/s) — moving it on-die is the gating problem at scale, GDDR tiering is not
  - gddr_mc perf counters DISABLED — GDDR RMW BW can only be inferred from wall-clock
  - wave overlap: mis-ordered semaphore handshake lets a band read stale grads (double-buffer aliasing) -> silently-wrong but plausible-looking loss; the S/T recurrence is a hard serial dependency within a tile's chunk stream

## Sequencing rationale

Ordered by leverage AND dependency, with a hard rule: build the missing primitive before anything that needs it. Stage 1 is pure host-side refactor (no kernel, no precision risk) so it banks 35-50ms immediately and in parallel with the harder work. Stage 2 (on-device reduce) is sequenced first among the architectural changes because the verdicts repeatedly showed it is the LOAD-BEARING prerequisite that does not exist today (m17:11 'reduce=host'): without it, hash-home would ship 32x32 tiles over the NoC (worse than today), drain-once would blow L1 (27MB/core), and the headline readback win is unreachable. Stage 3 then attacks the call-count residual that Stage 2 leaves (the readback is latency-bound, so bytes AND calls must both be cut — confirmed as the first exploration probe). Stage 4 (channel fusion) is a cheap multiplier that only pays once 2+3 collapsed the readback. Stage 5 (hash-home) is the architectural core but is correctly XL and placed after the reduce+drain substrate it depends on, with the owner-reduce de-risked by exploration probe 3 (and the key correction that FP32 drain is bit-exact, so int32 is optional, not required). Stage 6 makes accumulators resident and fuses projection — residency layer for scale. Stage 7 is explicitly the deferred endgame: the verdicts judged x280-in-loop, standalone barrier-removal/wave-overlap, and GDDR tiering as near-zero value at the measured N and/or dependent on everything prior, so they are bundled last and gated on justification (N at millions, or a need to remove the final host hop). The exploration track runs FIRST and de-risks the three XL bets (reduce precision, owner-reduce fan-out, persistent-kernel streaming) before any XL commitment.

## Risks & open questions

- Is the 260ms readback latency-bound or bandwidth-bound? Effective BW ~3GB/s suggests latency/call-count-bound, but this single number determines whether Stage 2 (bytes) or Stage 3 (calls) is the bigger lever. Resolve via exploration probe 1 before committing effort.
- Can an on-device reduce stay under the 2e-2 grad gate given m17 cannot enable fp32_dest_acc (all 16 bf16 dst slots used)? A 1024-term bf16 sum of signed products (a/b/c grads) is the main precision risk; enforce_fp32_accumulation in a separate reduce pass may rescue it. Probe 2.
- Does the m2 owner-single-writer pattern scale from 8src->1home to dense all-to-all without NoC wedge, and is FP32 drain fast enough to not be a new bottleneck? FP32 is bit-exact (m2_scatter_gather.py:53) so it is preferred; int32 fixed-point is a perf fallback that reintroduces quantization risk. Probe 3.
- Can a resident generic_op kernel loop chunks internally and read per-chunk params from L1 scratch (not runtime args, capped ~42 Gaussians)? The entire persistent-dataflow direction (Stages 3,5,6) rests on this being buildable on the ttnn stack. Probe 5; if it requires raw tt-metal Program/EnqueueProgram instead of generic_op, the effort estimates rise.
- Worst-case inbox depth under 10-25x replication: does it fit 1.5MiB/core L1 alongside CBs + raster shards at target N? Gates whether GDDR tiering (Stage 7b) is needed sooner than 2.4M Gaussians. Probe 4.
- Load imbalance: collapsing nbatch into one resident dispatch serializes the densest tile's full Gaussian stream on one core (worst at the synthetic nbatch=63). nbatch is geometry-bound (a Gaussian cannot move off its tile) so load-balancing cannot reduce it — the fix is the on-device reduce+drain, not redistribution. Confirm the hot-core serial floor does not erase the dispatch-collapse win.
- Bit-exactness vs the host fp64 scatter is lost by construction once any on-device reduce/accumulate is introduced; the acceptance bar shifts from bit-exact to the 2e-2 grad gate + multi-step loss-curve parity. Ensure the grad-equivalence harness (last_g3, device_resident.py:152) is the gate at every stage, and that a Stage-A-only error is localizable (it is end-to-end today).
- True usable core count + physical layout: g%ncores must map through a runtime device.compute_with_storage_grid_size lookup table, not the 140 descriptor template, because the worker grid has router-only column gaps. A wrong mapping is silent corruption or a NoC hang.
- x280 readiness: harts are still in reset and a prior bringup wedged NoC0. The on-die sort (Stage 7a) should first try an on-device counting-bucket before any x280 path, because x280-in-the-inner-loop risks wedging a single-owner training chip every step (recover only via tt-smi -r 0).
